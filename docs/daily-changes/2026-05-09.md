**2. Авто-сабмит OTP-кода после 6-го символа.**

**3. Привязка Telegram у email-first аккаунтов.**
- Миграция `worker/src/db/migrations/0027_users_tg_id.sql`:
  - `users.tg_id` (TEXT, UNIQUE) — для email-first юзеров стартует с
    NULL, выставляется при привязке. Для tg-first строк back-fill
    `tg_id := id`, чтобы новый lookup `findByTgId` находил их так же,
    как старый `findById`.
  - `tg_link_requests` (nonce / requester_id / tg_id / tg_username /
    tg_name / expires_at / created_at) — короткоживущая таблица для

- `worker/src/services/UserService.ts`:
  - `findByTgId(tgId)` — lookup по `tg_id`, fallback на `id`.
  - `upsert()` теперь ищет существующую строку через `findByTgId()` —

    а не плодит новую tg-строку (которую UNIQUE на `tg_id` всё равно
    отверг бы).
  - `linkTelegram(userId, tg)` — выставляет `tg_id`/`tg_username`/
    `tg_name`. Бросает `tg_id_taken`, если этот tg_id занят другим
    юзером.
- `worker/src/routes/auth.ts`: `isNew` теперь считается через
  `findByTgId()`, чтобы привязанный email-юзер не попадал под per-IP
  signup gate.
- `worker/src/routes/user.ts`:
  - `POST /user/me/telegram/link/start` — минтит nonce, кладёт строку
    в `tg_link_requests` с `requester_id = userId`. Возвращает nonce
    + `expiresAt`.
  - `GET /user/me/telegram/link/status/:nonce` — статус-поллер;
    `pending` / `expired` / `confirmed` / `conflict`. На `confirmed`
    зовёт `linkTelegram` и удаляет nonce-row (one-shot).
- `worker/src/bot/commands/start.ts`: добавлен ветка `args[0] ===
  'link_<nonce>'` — обновляет `tg_link_requests` с tg-данными
  отправителя и НЕ зовёт `ensureUser` (минимизирует риск UNIQUE
  conflict при последующем merge).
- `worker/src/cron.ts`: GC-сweep `DELETE FROM tg_link_requests WHERE
  expires_at < ?` рядом с `auth_nonces`-сweep'ом.
- `worker/src/routes/admin.ts`: при `DELETE /users/:id/data`
  чистится также `tg_link_requests WHERE requester_id = ?` (нет
  CASCADE FK).
- `src/components/features/LinkedAccountsPanel.tsx`: на ряде Telegram
  для email-first пользователя (`!hasTelegram && profile?.email`)
  показывается кнопка «Привязать Telegram». По клику зовёт
  `/user/me/telegram/link/start`, открывает
  `t.me/<bot>?start=link_<nonce>` в новой вкладке и поллит
  `/status/:nonce` каждые 1.5 c. На `confirmed` — `patchUser` +
  refetch профиля + toast. На `expired`/`conflict` — соответствующий
  toast и сброс состояния. Hard-stop через 5 минут.
- i18n: `profile.linkedAccounts.telegramLinkCta`,
  `telegramAwaiting`, `telegramLinkedToast`, `telegramExpired`,
  `telegramConflict`, `telegramStartFailed`. Добавлен `hero.moreActions`.

### Почему

- Hero на артисте/альбоме/плейлисте на узких viewport переносил

- OTP-форма требовала клика по «Подтвердить» после ввода 6 цифр —

- Email-first пользователи не имели способа добавить Telegram к
  аккаунту — пришлось бы регистрироваться заново через TG-flow и
  получать дубликат-аккаунт с потерянной библиотекой.

  предзаданным trigger'ом. Если захочется добавить новые pages с

  кебаба).
- `lib/share.ts/shareLink()` принимает `path` (например `/album/123`)
  и сам префиксит origin/base. Для трек-share уже есть
  `buildTrackShareUrl` + inline-логика в `TrackKebabMenu`/
  `ShareButton`; не дублируй, а зови этот хелпер.
- `submitCode()` в `EmailLoginCard` — async, но `useEffect`
  возвращает sync-функцию, поэтому используем `void submitCode(code)`.
  `eslint-disable react-hooks/exhaustive-deps` стоит намеренно: deps
  ограничены до `[code, step]`, потому что включение `submitCode` в
  массив re-fire'ило бы effect на каждом keystroke (функция
  пересоздаётся каждый рендер).
- Миграция 0027 ОБЯЗАТЕЛЬНО должна примениться до раскатки бекенда —
  `findByTgId()` упадёт без колонки `tg_id`. На staging выполняй
  `wrangler d1 execute … --file=worker/src/db/migrations/0027_users_tg_id.sql`
  до деплоя воркера.
- Деплой бота тоже обязателен: `start.ts` должен уметь обрабатывать
  `link_<nonce>` deeplink, иначе ссылка попадёт в обычный fallback

## Player every-other-track stutter (corsRetried symmetry fix only)

**Когда:** 2026-05-09 13:05 UTC (изменено 2026-05-09 14:50 UTC после реверта визуальных изменений)
**Кто:** Devin (по запросу пользователя)
**Branch:** `devin/1778331271-player-crossfade-strobe` (визуальная часть откачена в `devin/1778333137-revert-fullscreen-fx`)

### Что изменилось

**Чёт-нечётный обрыв кроссфейда (1→2 ок, 2→3 рвётся, 3→4 ок, 4→5 рвётся).**
- `src/hooks/useAudioPlayer.ts`:
  - `startCrossfade` non-preloaded path (после успешного
    `tryLoadSrc`): теперь выставляется `corsRetried[incoming] = true`.
  - `startCrossfade` preloaded promotion path: то же —
    `corsRetried[incoming] = true` сразу после промоушена.
  - Preload effect (фоновый префетч `queue[idx+1]`): после
    успешного `tryLoadSrc` — `corsRetried[incoming] = true`.
- Корень: `loadTrack` (только initial-track path) выставляет
  `corsRetried[slot] = true` после успешной загрузки, чтобы
  потенциальный mid-stream `error` в дальнейшем НЕ дёргал
  `reloadWithoutCors()` (который зовёт `audio.load()` и
  принудительно сбрасывает `currentTime` в 0 + ставит на паузу).
  Все остальные пути загрузки (preload + промоушен через
  кроссфейд) этот флаг НЕ выставляли. Из-за этого слот A после
  стартового `loadTrack(t1)` стабильно имел `corsRetried[a] = true`,
  а слот B (никогда не проходивший через `loadTrack`) — постоянно
  `corsRetried[b] = false`. На любую транзиентную ошибку
  (range-request 4xx, тайм-аут, плавающий CORS-заголовок Tidal-
  CDN) на слоте B вызывался `reloadWithoutCors`, который
  обрывал ramp/паузил аудио, что воспринималось как «пауза, без
  плавного перехода, иногда стопает».
- Слот A был активен на нечётных треках (1, 3, 5, …), слот B —
  на чётных (2, 4, 6, …). Поэтому пользователь видел паттерн:
  «1→2 ок, 2→3 ломается, 3→4 ок, 4→5 ломается».

### Что отменено (относительно ранее зафиксированных правок)

- **Bass-driven white strobe** на блюр-фоне fullscreen-плеера
  (PR #422, FullscreenPlayer.tsx) — пользователь попросил
  откатить.
- **Cover-tinted strobe** + slow drift на блюр-фоне (PR #423) —
  откачено вместе с визуалом из #422.
- **`useBassPulse(active, resetKey?)` + `ctx.resume` в RAF tick**
  (PR #423, useAudioPlayer.ts) — откачено, сигнатура хука
  возвращена к оригинальной `useBassPulse(active)`. Существующий
  halo-эффект на TiltCard продолжает работать как раньше.

### Почему откатили

Пользовательский запрос: «сделай реверт до момента как было в
PR #421, только оставь момент фикса, связанный с every-other-track
stutter on auto-crossfade». То есть оставить только функциональный
fix кроссфейда, а все визуальные эксперименты в fullscreen-плеере
вернуть к состоянию #421.

### Что важно знать следующему агенту

- `corsRetried` — это «бюджет на одну CORS-перезагрузку слота
  на src». После любого успешного `tryLoadSrc` бюджет должен
  гасить КАЖДЫЙ путь, который этот src туда положил. Сейчас
  в коде четыре таких пути: (a) `loadTrack`, (b) preload effect,
  (c) `startCrossfade` non-preloaded, (d) `startCrossfade`
  preloaded promotion. Если появится пятый — обязательно
  ставить `corsRetried[slot] = true` сразу после успеха.
- FullscreenPlayer возвращён к состоянию #421: блюр-кавер на
  static image, halo с pulse+flash от `useBassPulse`, тёмный
  градиент 15/30/50, никаких дополнительных слоёв.


## Offline lyrics — `devin/1778362098-offline-lyrics`

**Когда:** 2026-05-09 21:30 UTC
**Кто:** Devin (по запросу пользователя)
**Branch:** `devin/1778362098-offline-lyrics`
**PR:** см. PR в репозитории

### Что изменилось

- `src/lib/offline/types.ts`:
  - Новый интерфейс `OfflineLyrics` (зеркалит `LyricsResponse` от
    `/tracks/:id/lyrics` плюс `fetchedAt`).
  - В `OfflineTrack` добавлено опциональное поле `lyrics: OfflineLyrics`.
- `src/lib/offline/streamResolver.ts`: добавлена функция
  `fetchLyricsPayload(trackId, source)` — best-effort fetch лирики,
  никогда не бросает (ошибки → null), пропускает upload-треки.
- `src/lib/offline/downloads.ts :: runTrack`: лирика запрашивается
  параллельно с `fetchAudioBlob` (нулевая прибавка к wall-clock),
  записывается в IDB одной транзакцией с аудио-блобом. Никогда не
  затирает существующую офлайн-лирику пустым ответом.
- `src/lib/offline/index.ts`: экспортируем `fetchLyricsPayload` и
  тип `OfflineLyrics` через публичный API модуля.
- `src/hooks/useLyrics.ts`: новая стратегия — IDB-first, потом сеть,
  на сетевой ошибке re-check IDB; при успешной сетевой загрузке
  back-fill офлайн-строки если у неё ещё нет лирики (для треков,
  скачанных до этой фичи).

### Почему

Пункт 1 батча 2026-05-09: «при загрузке трека в офлайн режим
прогружать его lyrics, чтобы они работали и в офлайн режиме (PWA
точно чтоб работало)». До этого PR `useLyrics` всегда ходил в
