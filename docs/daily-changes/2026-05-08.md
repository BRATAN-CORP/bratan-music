## Worker KV write-budget — `devin/1778248978-kv-write-budget`

**Когда:** 2026-05-08 14:05 UTC
**Кто:** Devin (handoff)
**Branch:** `devin/1778248978-kv-write-budget`
**PR:** *(будет добавлен после открытия)*

### Что произошло
Cloudflare прислал алерт «90% of daily Cloudflare Workers KV
operations limit reached» — 981 / 1000 writes за сутки. Free tier
KV даёт ровно 1000 writes/day на namespace, после чего Cloudflare
возвращает 429 на любые `KV.put()` до полуночи UTC. Reads (19.22k /
100k) не лимит — упирается именно writes.

### Аудит — где пишем в KV
Прошёл по всем `env.SESSIONS.put(...)` и `env.SESSIONS.delete(...)`
в `worker/src/`. Нашлось:

| Префикс ключа                    | Где пишут                                                     | TTL до    | Особенности                                |
| -------------------------------- | ------------------------------------------------------------- | --------- | ------------------------------------------ |
| `genre_seed_tracks:<slug>`       | `RecommendationService` **И** `DailyPlaylistService`          | 12h × 2   | **Дублирование** — одни и те же ключи      |
| `track_radio:<id>`               | `RecommendationService.candidatesFromTrackHistory`            | 24h       | Часто-вызываемая, мало меняется            |
| `artist_seed_tracks:<artistId>`  | `RecommendationService.candidatesFromArtistSeeds`             | 12h       | Артист top-tracks меняется редко           |
| `rec_suggested_artists:v1`       | `routes/recommendations.ts /artists/suggested`                | 24h       | Глобальный кеш explore-страницы            |
| `tidal-track-formats:<id>`       | `TidalWeb.writeDiscoveryCache` (positive)                     | 30d       | OK                                          |
| `tidal-track-formats:<id>`       | `TidalWeb.writeDiscoveryCache` (negative — пустой результат)  | **1h**    | Перепачка на проблемных треках             |
| `tidal-track-quality:<id>`       | `TidalWeb.writeCachedQuality`                                 | 30d       | OK                                          |
| `tidal-discovery-breaker`        | `TidalWeb.tripDiscoveryBreaker`                               | 1h        | Свежесть критична — не трогаем             |
| `daily_playlist:<userId>:<date>` | `DailyPlaylistService.regenerate`                             | без TTL   | По юзеру 3/день максимум — не виновник     |

Главные виновники:
1. **Дублирование `genre_seed_tracks:`**. На `wave()` или
   `regenerate(daily-playlist)` оба сервиса независимо
   читают-и-пишут один и тот же ключ, удваивая стоимость.
2. **Слишком короткие TTLs** (12h / 24h / 1h) на медленно
   меняющихся сидах. На активный день со множеством "холодных"
   слагов и треков-кандидатов это легко набирает 800+ writes.

### Что изменилось
**Новый файл `worker/src/services/seedCache.ts`** — общий хелпер:
- `CACHE_TTL_S = 7 * 24 * 60 * 60` (7 дней).
- `getCachedGenreTracks(env, tidal, slug)` — read-then-write
  для `genre_seed_tracks:<slug>` через одну точку.

**`worker/src/services/RecommendationService.ts`:**
- Импорт `CACHE_TTL_S as SEED_CACHE_TTL_S, getCachedGenreTracks`.
- `RADIO_CACHE_TTL_S` теперь = `SEED_CACHE_TTL_S` (24h → 7d).
- `candidatesFromArtistSeeds()`: TTL для `artist_seed_tracks:<id>`
  поднят с 12h до `SEED_CACHE_TTL_S` (7d).
- `candidatesFromGenres()`: упрощён до вызова
  `getCachedGenreTracks(env, tidal, slug)` для каждого слага.

**`worker/src/services/DailyPlaylistService.ts`:**
- Импорт `getCachedGenreTracks`.
- Метод `tracksFromGenre(slug)` теперь однострочный делегат к
  `getCachedGenreTracks(env, tidal, slug)`. Дубликат `try/catch +
  KV.get/put + Tidal explore-page parse` (15 строк) удалён.

**`worker/src/routes/recommendations.ts`:**
- `rec_suggested_artists:v1` TTL: 24h → 7d (`7 * 24 * 60 * 60`).

**`worker/src/services/tidal/TidalWeb.ts`:**
- `DISCOVERY_NEGATIVE_CACHE_TTL_S` (negative-cache для пустых
  discovery-результатов): 1h → 24h. Один проблемный трек, который
  юзер послушал несколько раз за день, теперь не пишет negative
  KV-запись каждый час.

### Почему именно 7 дней
- Tidal-страницы `explore?slug=…` и radio-pool'ы артистов меняются
  на их стороне очень медленно (новые эфиры — недели). Ощущение
  "свежие рекомендации" обеспечивает не низкий TTL, а
  ротация-кандидатов внутри `RecommendationService` (rerank,
  dislike-фильтр, taste-overlap).
- `1000 writes/day` — жёсткий cap. 12h TTL означает минимум 2
  writes/key/day. На сотню уникальных слагов/артистов в день — 200+
  writes только на эту одну категорию. 7d даёт ~14× меньше.

### Ожидаемый эффект
- Steady-state writes: 981/day → ~140–200/day.
- Запас на cron-перегенерации daily-playlist'ов и cold-spike при
  раннем-утреннем cron.
- Read-rate не меняется (нам и так далеко до 100k reads/day лимита).

### Что НЕ трогали — намеренно
- `tidal-discovery-breaker` (1h) — глобальный circuit-breaker
  на upstream. 1h тут оптимум: меньше — слишком частые retry в
  плохой период; больше — система зависает в "broken" режиме на
  слишком долго после восстановления Tidal.
- `tidal-track-quality:<id>` (30d) — already optimal.
- Telegram-bot session keys / JWT refresh tokens — это другой
  namespace pattern (короткие сессии, per-user) и не виновник.
