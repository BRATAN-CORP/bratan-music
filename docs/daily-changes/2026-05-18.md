`worker/src/services/tidal/TidalApi.ts`:
- Новый `commonParams(extra)` хелпер пробрасывает
  `countryCode` + `locale` + `deviceType=BROWSER` +
  `includeExplicit=true` + `explicitContent=true` через каждый
  catalogue endpoint: `search`, `getTrack`, `getAlbum`,
  `getAlbumTracks`, `getArtist`, `getArtistTopTracks`,
  `getArtistAlbums`, `getSimilarArtists`, `getTrackRadio`,
  `getArtistRadio` (включая mix-fallback), `getPlaylistTracks`,
  `getPage`, `getArtistPage`, `getPageData`. `useEditedLyrics=false`
  оставлен только в lyrics fallback chain (он специфичен для
  lyrics endpoint, на остальных Tidal иногда строго валидирует
  unknown params).
- HTTP-headers переписаны под Tidal Web client:
  `User-Agent: Mozilla/5.0 ... Chrome/124.0.0.0` +
  `Origin: https://listen.tidal.com` +
  `Referer: https://listen.tidal.com/` +
  `Accept-Language: en-US,en;q=0.9`. **Это критично**: Tidal
  honor'ит request-level `includeExplicit/explicitContent`
  overrides только когда запрос выглядит как Web client; mobile
  UA (раньше) даёт ignore-overrides+fallback на per-account фильтр.
  Это объясняет основной симптом «в самом Tidal с того же
  аккаунта без цензуры, у меня — с цензурой».

`worker/src/services/tidal/TidalService.ts`:
- `normaliseAlbumTitle` расширен: regex теперь strip'ает
  `clean|explicit|edited` токены через
  `\s*[([][^()\[\]]*\b(?:deluxe|expanded|remastered|anniversary|extended|special|edition|version|bonus|reissue|clean|explicit|edited)\b...`,
  обёрнутый в `\b(?:...)\b`. False-positives типа
  `(Cleaning Up Mix)`, `(Cleaner Cut)`, `(Explicitly Mine)`,
  `(Editions)` НЕ collapse'ятся (whole-word boundary), а
  легитимные `(Clean)` / `[Explicit]` / `(Edited Version)`
  собираются в один bucket с explicit-twin'ом.
- **NEW: `preferExplicitAlbums(items)`** — same-response dedupe.
  Группирует по `(artistId, normaliseAlbumTitle(title))`; если
  и clean, и explicit twin есть в одном ответе, drop'ает clean.
  Order-preserving (single-pass record + filter).
- **NEW: `preferExplicitTracks(items)`** — то же для треков.
  Группирует по `(artistId, normaliseAlbumTitle(title), durationBucket)`
  где `durationBucket = round(duration / 2)` — collapse'ит
  3:14 vs 3:15 (метаданные различаются на 1с) но не разные
  миксы. Sentinel keys для треков без artistId/duration —
  никогда не collapse'ятся false-positively.
- **NEW: `swapInExplicitTwins(items)`** — **это ключевой
  missing piece**, которого не было в PR #456. Для каждого
  clean трека делает active per-track Tidal-search lookup с
  query `<title> <artistName>`, фильтрует кандидатов по
  `cand.explicit === true && cand.artistId === clean.artistId
   && abs(cand.duration - clean.duration) <= 2 &&
   normaliseAlbumTitle(cand.title) === normaliseAlbumTitle(clean.title)`.
  Hydrate'ит найденный twin через `getTrack(twinId)` и
  substitute'ит на тот же индекс в исходном массиве.
  Concurrency=4 in-flight (CF Worker subrequest budget),
  KV-memo `tidal-explicit-twin:<cleanId>` →
  `<explicitId>` (positive) или `__none__` (negative) на 30
  дней. Это **решает кейс**, который PR #456 не закрывал:
  «Tidal вообще не вернул explicit вариант в search response» —
  same-response dedupe там не помогает, потому что нечего
  collapse'ить; нужен active follow-up lookup.
- Wired:
  - `search()` — `preferExplicitTracks` + `preferExplicitAlbums`
    + `swapInExplicitTwins` на tracks bucket.
  - `getArtistTopTracks()` — после фильтра по primary credit:
    `preferExplicitTracks` + `swapInExplicitTwins`.
  - `getPlaylistTracks()` — `preferExplicitTracks` +
    `swapInExplicitTwins`.
  - `getArtistAlbumsAndSingles()` — `preferExplicitAlbums` на
    обоих split'ах (albums + singles).
  - `getArtistReleases()` — `preferExplicitAlbums` ПЕРЕД
    `dedupeAlbums` (Tier-2 collapser), чтобы explicit twin'ы
    survival'или title-fingerprint dedupe.
  - `getArtistAlbums()` — `preferExplicitAlbums`.
- `getAlbum()` НЕ swap'ит и НЕ redirect'ит. Тут id = контракт
  пользователя; если он явно вставил clean-edition id — мы
  не second-guess'им. Только `console.warn` breadcrumb на
  clean-suffix titles так что operator может увидеть в
  `wrangler tail`. Фикс «search возвращает clean-edition
  альбом» происходит на уровне выше — в `search()`, где
  `albums[]` прогоняется через `preferExplicitAlbums`.

### Что важно знать следующему агенту

- **Active twin lookup стоит небольшие deyalty**: каждый clean
  трек в search/topTracks/playlistTracks → одна follow-up
  Tidal-search query (≤4 параллельно). На cold cache это
  ~200-400 ms на response с большим количеством clean треков.
  KV-memo на 30 дней нивелирует это после первого хита. Если
  обнаружится upstream rate-limiting — concurrency можно снизить
  до 2.
- **Negative cache (`__none__`)**: для треков, у которых
  действительно нет explicit twin'а (clean-only releases),
  негативный результат закешируется на 30 дней. Если Tidal
  потом залив explicit twin — мы об этом не узнаем до истечения
  TTL. Это приемлемый trade-off (catalogue races редкие).
- **Пользовательская жалоба об «остался гэп между названием и E»
  в mini-player** была частью отдельного UI-pass'а: gap-1 даёт
  ~4 px, что соответствует Apple Music/Spotify. Если хотят ещё
  плотнее — gap-0.5 (~2 px) можно попробовать, но риск что E
  визуально «слипнется» с последней буквой названия.
- **getAlbum() намеренно не делает swap**. Если пользователь в
  будущем попросит «и в getAlbum тоже подменяй на explicit twin
  по умолчанию» — это можно добавить, но это меняет контракт
  (ID, который ввели = ID, который играет). Сейчас warn
  breadcrumb пишется в `wrangler tail` для clean-suffix titles —
  это видно в health logs.
